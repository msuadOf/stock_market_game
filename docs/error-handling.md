# 错误处理哲学 (Error Handling)

> 这是项目最鲜明的工程立场：**错误必须显式可见，绝不静默吞掉。**
> 配套：[`principles.md`](principles.md) 原则 2。

---

## 0. 核心立场

> **一个会崩溃并告诉你为什么的程序，远好于一个不崩溃但行为错误的程序。**

对游戏尤其如此：静默 fallback 会导致——存档损坏、资金凭空消失、市场数据错乱、持仓数字对不上。
这些 bug 比"弹一个错误框"**严重得多**，而且极难定位。

因此本项目的默认策略是 **fail loud（大声报错）**，而不是 fail silent（静默兜底）。

## 1. 红线（禁止的做法）

### ❌ 禁止：空 catch / 吞掉错误

```ts
// 坏：错误被吞，问题永远隐藏
try {
  saveGame(state);
} catch {
  // 什么也不做
}

// 坏：返回 null 把错误伪装成"正常空值"
function loadPrice(): number | null {
  try { return fetchPrice(); } catch { return null; }
}
```

### ❌ 禁止：用默认值掩盖异常

```ts
// 坏：如果 parsedPrice 是 NaN（解析失败），悄悄用 0 —— 资金凭空归零
const price = Number(json.price) ?? 0;
const price = isNaN(Number(json.price)) ? 0 : Number(json.price); // 同样坏：掩盖了"数据本不该是脏的"
```

### ❌ 禁止：catch 后只 `console.log`

```ts
// 坏：开发者看到了，用户看不到；问题被"记一笔"然后放任继续
catch (e) { console.error(e); }
```

## 2. 应该怎么做

### ✅ 抛出带丰富上下文的错误

```ts
// 好：带上"发生了什么 / 在哪 / 输入是什么 / 为什么"
throw new GameError({
  code: 'SAVE_CORRUPTED',
  message: '存档文件已损坏，无法读取持仓数据。',
  where: 'SaveModel.load',
  cause: err,
  context: { saveId, rawSnippet: raw.slice(0, 200) },
  // UI 据此渲染：标题、描述、可复现细节、上报按钮
});
```

### ✅ 让错误显式到达用户

- **UI 层必须渲染错误**，而不是让它在控制台里腐烂。
- 错误展示包含：**发生了什么**（人话）、**在哪发生的**、**用户能做什么**（重试 / 反馈 / 重新开始）、**可复现细节**（供反馈）。
- 见下方 §4 的错误展示契约。

### ✅ 用可辨识的结果类型表达"可预期的失败"

对于**业务上可预期**的失败（买入资金不足、停牌不能交易），不要用异常，用**显式的 Result 类型**：

```ts
type Result<T, E> = { ok: true; value: T } | { ok: false; error: E };

// 好：调用方被迫处理"买不起"这个分支，不会被静默吞掉
function buy(state, order): Result<TradeOutcome, TradeError> { ... }
```

> 区分两类失败：
> - **可预期的业务失败**（资金不足、停牌）→ Result 类型，调用方显式处理。
> - **不可预期的错误**（存档损坏、不变量被违反、NaN）→ 抛错 + UI 显式展示。

## 3. 何时可以 fallback（少数例外）

只有同时满足以下条件，才允许"失败时用默认值/降级"：

1. 失败是**真正非致命**的（如：某个可选的装饰性配置缺失）。
2. 默认值是**用户明显期望且无副作用**的。
3. **仍然记录/展示**这次降级（不让它完全静默）。

> 默认是**不 fallback**。要用 fallback 必须在 PR 里论证它满足上述三条。
> 这是"减少规避错误的 fallback"原则的落地。

## 4. 错误展示契约（面向 UI）

UI 收到一个错误时，至少要能渲染以下字段（来自错误对象）：

| 字段 | 用途 | 示例 |
|------|------|------|
| `code` | 机器可读的错误码，便于分类/搜索 | `SAVE_CORRUPTED` |
| `message` | 给用户的人话描述 | "存档已损坏，无法继续游戏。" |
| `where` | 发生位置（层 + 模块 + 函数） | `SaveModel.load` |
| `cause` | 原始错误（链式） | 解析异常对象 |
| `context` | 可复现细节（脱敏） | saveId、输入片段 |
| `recoverable` | 是否可恢复 + 建议动作 | "可重试 / 可上报 / 需重启存档" |

**用户视角的目标**：当他点"反馈/上报"时，能一键复制一段**足够复现**的错误详情——
这正是本项目"方便反馈和修改"原则的落点。

## 5. 输入校验（防御式编程的日常）

对所有**外部输入**，在边界处校验：

| 输入源 | 校验时机 | 失败处理 |
|--------|----------|----------|
| 用户输入（表单） | 提交时 + UI 即时反馈 | 字段级错误提示 |
| 存档文件 / LocalStorage | 读取时 schema 校验 | 抛 `SAVE_CORRUPTED` + UI 展示 |
| 网络响应 | 反序列化时校验 | 抛 `BAD_RESPONSE` + 可重试 |
| 命令行 / 配置 | 启动时校验 | 启动失败 + 明确原因 |

> 推荐用 schema 校验库（如 Zod）在边界做运行时校验。具体库以 ADR 为准。

## 6. 不变量与断言

核心逻辑里应显式声明**不变量**（如：现金 + 持仓市值 == 总资产），并用断言守护。

- 不变量被违反 = 程序进入未定义状态 → **立即抛错 + 展示**，不要试图"修复"它（修复本身就是掩盖）。
- 断言失败的信息要足够定位：哪个不变量、实际值、期望值。

## 7. 一句话总结

> **可预期的失败用 Result 显式表达；不可预期的错误抛出带上下文的异常并由 UI 展示给用户。
> 永远不在静默中 fallback。让错误成为改进的信号，而不是被掩埋的炸弹。**
