# 工程原则 (Engineering Principles)

> 本文档是项目的工程宪章。所有代码（无论人类还是 AI 写的）都应能通过这些原则的检验。
> 配套阅读：[`testing.md`](testing.md)、[`error-handling.md`](error-handling.md)、[`architecture.md`](architecture.md)。

---

## 原则 1：测试驱动开发（TDD）

**先有测试，后有代码。** 测试不是事后的质量保障，而是设计的工具。

- 红 → 绿 → 重构，每一步都是小提交。
- 写实现前，先用测试把"正确行为"描述清楚——这逼你想清楚接口和边界。
- 公共 API、核心业务逻辑（撮合、价格模型、组合计算）**必须有**测试。
- 详见 [`testing.md`](testing.md)。

**反模式：** "先把功能写完，回头补测试。" —— 这不是 TDD，且几乎总会产生"为代码找借口"的弱测试。

## 原则 2：防御式编程 + 显式错误

**代码应该在不合法的状态下大声叫喊，而不是悄悄苟活。**

- 对**所有外部输入**（用户输入、网络响应、存档文件、LocalStorage、命令行参数）做校验。
- 校验失败 → 抛出/返回**带上下文的错误**，由 UI 层显式展示给用户。
- 禁止静默 fallback（`catch { return null }`、`x ?? 默认值` 掩盖 `undefined` bug）。
- 详见 [`error-handling.md`](error-handling.md)。

**核心立场：** 一个**会崩溃并告诉你为什么**的程序，远好于一个**不崩溃但行为错误**的程序。
对于游戏，错误行为意味着存档损坏、资金凭空消失、市场数据错乱——这比一次显式报错糟糕得多。

## 原则 3：关注点分离（核心 = 纯逻辑，外壳 = 副作用）

游戏的核心（规则、模拟、计算）必须是**纯函数 + 可序列化状态**，与渲染、网络、存储、定时器完全解耦。

- `engine` 层：不知道 React、不知道 DOM、不知道网络。给它状态 + 输入，返回新状态。
- `web` / `server` / `desktop`：负责 I/O、渲染、把用户的动作翻译成对 engine 的调用。
- 这样核心可以被：前端直接跑 / 后端跑 / Tauri 里跑 / 测试里跑——**同一份代码**。

> 这是"后端可选"能成立的地基。详见 [`architecture.md`](architecture.md)。

## 原则 4：单一数据访问层（Single Source of Truth）

任何持久化数据（存档、设置、市场状态）只通过**一个**受控的接口读写。

- 避免散落在各处的 `localStorage.getItem` / 文件直接读写。
- 所有读写都经过校验 + 显式错误（呼应原则 2）。
- 这让"换存储后端"（LocalStorage → IndexedDB → 服务端）成为局部改动。

## 原则 5：最小惊讶 & 显式优于隐式

- 函数 / 组件只做名字承诺的事，副作用要显式（命名上体现，如 `saveGame()` 而非 `update()`）。
- 避免"魔法"：隐式全局状态、隐式类型转换、隐式重试。能用显式就用显式。
- 类型要诚实：宁可 `Result<T, E>` / 可辨识联合，也不要 `T | null` + 调用方靠猜。

## 原则 6：小步前进，可回滚

- 一次改动解决一个问题。
- 小提交、小 PR，便于评审和回滚。
- 重构与功能变更**分开提交**（不要在 `feat` 里偷塞重构）。

## 原则 7：文档与代码同等重要

- 架构决策写进 ADR（[`decisions/`](decisions/)），而不仅仅是代码注释。
- 改动影响架构 / 约定 → 必须同步更新文档。
- 注释解释 **why**，不解释 what（what 看代码就行）。

## 原则 8：性能是约束，不是过早优化

- 先写**正确且可读**的代码，让测试覆盖它。
- 只有在测试 / 剖析证明某处是瓶颈时，才针对性优化——且优化后测试依然全绿。
- 游戏循环（市场 tick）关注性能，但不在没有数据的情况下盲猜优化。

---

## 一句话总结

> **先用测试说清楚要什么 → 用纯逻辑实现它 → 对一切外部输入防御式校验 → 出错就大声说 → 保持小步、显式、可回滚。**
